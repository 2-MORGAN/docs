## Pourquoi un PAT ?

CNP utilise votre Personal Access Token (PAT) GitLab pour agir en votre nom : lire vos projets, pousser des fichiers de configuration, créer des deploy tokens, protéger des branches et déclencher des pipelines.

Le token est stocké **chiffré (AES)** dans la base PostgreSQL de CNP — il persiste au redémarrage du backend.

---

## Créer un PAT sur GitLab

<Steps>
  <Step title="Accéder aux paramètres">
    Connectez-vous à GitLab, puis allez dans **Profile → Preferences → Access Tokens**.
  </Step>
  <Step title="Créer un nouveau token">
    Cliquez sur **Add new token** et remplissez :

    | Champ | Valeur recommandée |
    | --- | --- |
    | Token name | `cnp-platform` |
    | Expiration date | Dans 1 an maximum |
    | Scopes | `api`, `write_repository` |

    <Warning>
      La **date d'expiration est obligatoire** lors de l'enregistrement dans CNP. Notez-la.
    </Warning>
  </Step>
  <Step title="Copier le token">
    Copiez le token généré — il ne sera affiché qu'une seule fois.
  </Step>
</Steps>

### Scopes requis

| Scope | Raison |
| --- | --- |
| `api` | Lecture/écriture de projets, variables CI, branches protégées, deploy tokens, pipelines |
| `write_repository` | Commit de fichiers de scaffold sur le dépôt |

---

## Enregistrer le PAT dans CNP

1. Connectez-vous à la plateforme CNP
2. Si le PAT n'est pas configuré, une bannière jaune s'affiche
3. Cliquez sur **Configurer le token GitLab**
4. Remplissez le token **et la date d'expiration** (format `YYYY-MM-DD`)
5. Cliquez sur **Enregistrer**

### Erreurs possibles

| Erreur | Cause | Solution |
| --- | --- | --- |
| `Token GitLab invalide` | Token expiré ou incorrect | Générez un nouveau PAT |
| `la date d'expiration du token est requise` | Champ `expires_at` manquant | Renseignez la date d'expiration |
| `format de date d'expiration invalide` | Format incorrect | Utilisez `YYYY-MM-DD` |
| `impossible de joindre GitLab` | Backend ne peut pas contacter GitLab | Vérifiez `GITLAB_BASE_URL` |

---

## Persistance

Contrairement aux versions précédentes (store in-memory), le PAT est désormais **persisté dans PostgreSQL** avec le token chiffré. Il survit aux redémarrages du backend.
