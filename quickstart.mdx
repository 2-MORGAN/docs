## Prérequis

- Un compte GitLab avec un dépôt existant
- Un [Personal Access Token (PAT)](/guides/pat-setup) GitLab avec les scopes `api` et `read_user`
- Accès à l'instance CNP fournie par votre administrateur

---

## Étape 1 — Connexion

Rendez-vous sur l'URL de la plateforme CNP. Deux méthodes de connexion sont disponibles :

<Tabs>
  <Tab title="Microsoft SSO (recommandé)">
    Cliquez sur **Se connecter avec Microsoft** et authentifiez-vous avec votre compte `@epita.fr`.

    CNP utilise MSAL pour obtenir un ID Token Azure AD, qui est ensuite validé côté backend. Seuls les comptes du domaine autorisé (`epita.fr`) sont acceptés.
  </Tab>
  <Tab title="Compte admin local">
    Saisissez les identifiants admin fournis par votre administrateur :

    ```text
    Username: <fourni par l'admin>
    Password: <fourni par l'admin>
    ```
  </Tab>
</Tabs>

<Note>
  Le token JWT est valide 24h et stocké dans le localStorage. Il est identique quelle que soit la méthode de connexion utilisée.
</Note>

---

## Étape 2 — Configurer votre PAT GitLab

Après la connexion, une bannière s'affiche si votre PAT GitLab n'est pas encore configuré.

1. Cliquez sur **Configurer le token GitLab**
2. Collez votre PAT dans le champ
3. Cliquez sur **Enregistrer**

CNP valide immédiatement le token en appelant l'API GitLab. Si le token est valide, votre nom d'utilisateur GitLab s'affiche.

<Tip>
  Voir [Comment créer un PAT GitLab](/guides/pat-setup) pour les instructions détaillées.
</Tip>

---

## Étape 3 — Configurer un service

1. Depuis le **Dashboard**, cliquez sur **Nouveau service**
2. Dans la modale, sélectionnez le dépôt GitLab cible dans la liste déroulante
3. Choisissez le framework correspondant à votre projet :
   - `go` pour un service Go
   - `nextjs` pour une application Next.js
   - `springboot` pour un service Spring Boot
4. Cliquez sur **Configurer**

<Warning>
  CNP va pousser des fichiers directement sur la branche par défaut de votre dépôt (`main` ou `master`). Assurez-vous que votre branche est propre avant de continuer.
</Warning>

---

## Étape 4 — Vérifier le pipeline

CNP a automatiquement :

- Poussé le `.gitlab-ci.yml` et le `Dockerfile` (\+ fichiers de config selon le framework)
- Configuré la protection de la branche principale
- Injecté les variables CI (`REGISTRY_TOKEN`, `REGISTRY_URL`, `REGISTRY_USERNAME`)

Rendez-vous dans votre projet GitLab → **CI/CD → Pipelines** pour voir le pipeline en cours.

---

## Étape 5 — Redéployer (optionnel)

Pour forcer un nouveau déploiement depuis le dashboard :

1. Trouvez votre service dans la liste
2. Cliquez sur **Redéployer**

CNP re-génère les fichiers de scaffold et déclenche un nouveau pipeline sur la branche par défaut.

---

## Résultat

Votre service est maintenant :

<Steps>
  <Step title="Scaffoldé">
    Les fichiers de CI/CD et Docker sont présents sur votre dépôt
  </Step>
  <Step title="Taggé">
    Les topics GitLab `cnp-2morgan`, `cloud-native` et votre framework sont ajoutés au projet
  </Step>
  <Step title="Protégé">
    La branche principale est protégée (push désactivé, merge ouvert aux Developers)
  </Step>
  <Step title="Prêt à builder">
    Le pipeline démarre automatiquement à chaque push sur `main`
  </Step>
</Steps>
